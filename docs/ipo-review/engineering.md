# IPO Panel — Engineering, Reliability & Security Review

**Scope:** `server.js` (798 lines, read in full), `public/index.html` (2,955 lines), `e2e.js`, `e2e-playwright.js`, `qa-*.js`, `package.json`.
**Lens:** diligence-grade scrutiny for multi-room, hostile-network use. All fixes respect the project invariant: **no framework, no build step**; vendoring small plain-file libs is the only pressure valve.
**Date:** 2026-06-13.

What's already right (credit where due): hostKey-gated Farrier actions are consistently enforced (`isFarrier()` on every privileged WS message); locked-block re-assertion + re-injection on `redesign:update` is genuinely server-side; the WS heartbeat (30s ping/pong + terminate) follows the canonical `ws` pattern; the client's `esc()`/text-node discipline held up under audit (block labels, chat bubbles, team names all flow through `createTextNode` or `esc()` — no live XSS found); client re-render interval hygiene is clean (`render()` clears lobby/tour/poster timers, index.html:1014–1017); `/api/coach` degradation is honest (200 + `degraded:true`). The gaps below are about what a hostile or unlucky network does to that foundation.

## 2. Test gaps — the 5 most valuable missing tests

The existing suites are strong on the *happy-path contract* (34 WS checks incl. one lock-tamper case, 64 browser UAT checks, 20 fix regressions). What's untested is exactly what diligence cares about:

1. **Hostile-payload suite** (extends `e2e.js`): oversized canvas (5 MB blocks array), `x:NaN`/`w:"<style>"` geometry, `locked:true` forgery on own blocks, `peopleLandings` overwrite via `redesign:update` (finding #5 — currently passes, i.e. the bypass works), `chat:post role:'system'` spoof, `assumption:resolve` from the wrong team, `lock:challenge field:'personas'` → approve. Each asserts the server clamps/rejects **and stays responsive** (follow with a ping round-trip under 200 ms).
2. **Kill-and-restore mid-phase**: drive to mid-Rebuild, `SIGKILL` the server (a) between debounce and write, (b) mid-write (truncate the file artificially), restart, reconnect all actors — assert state survives or degrades to the `.bak`, locks still enforced, reveal not replayed. Persistence is the headline ✅ in CLAUDE.md and has **zero** automated coverage of the failure half.
3. **Concurrent-edit clobber**: two member sockets commit different new blocks within one round-trip window; assert both blocks exist afterwards. This codifies finding #11 — today it fails, which is the point: it pins the known LWW limitation and catches regressions once block-merge ships.
4. **Reconnect/presence storm**: 12 clients × 10 rapid disconnect/reconnect cycles (some via `terminate` to exercise the heartbeat path, some mid-`team:join` with `reclaimMemberId`); assert no ghost members, correct `online` flags, exactly one seat per human, and server RSS stable.
5. **Authorization matrix sweep**: programmatically fire every farrier-gated message type (`phase:set`, `timer:*`, `member:remove`, `member:reseat`, `team:remove`, `present:set`, `teardown:regenerate`, `lock:resolve`) from a member socket and every member message from a pre-join socket; assert uniform rejection. Today only `lock:resolve` has this test — the others are enforced in code but unpinned.

## 3. Verdict

**The single most important engineering investment before real-world (multi-room, hostile-network) use is a server-side input/authorization hardening pass — roughly 150 lines: `maxPayload` + canvas sanitization, whitelist-merge on `redesign:update`, per-socket rate limiting, coach-proxy gating, and atomic persistence with shutdown flush.** The product's locked-methodology rules (#4, #6, #8) are currently enforced for honest clients but trivially bypassed or stalled by one dishonest device holding the projected room code. Nothing here needs a framework, a build step, or even a new dependency — it is the cheap, boring 1% that makes the clever 99% safe to put in a room full of strangers.
